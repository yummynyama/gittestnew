# Как подогнать шаблон под любое задание экзамена

Этот файл — **пошаговая инструкция**, если в задании не «Корочки.есть», а другая фирма, другие поля в базе или вообще другая сфера (отель, автосервис, библиотека и т.д.).

> **Главная мысль:** шаблон — это **каркас**, а не готовая работа «на все случаи».  
> На экзамене вы **читаете новое ТЗ** и меняете проект под него. Большинство заданий ДЭ устроены **одинаково по структуре** — меняются только названия, поля и предметная область.

## 1. Подходит ли шаблон под каждое задание?

### Да, если в задании есть похожий функционал:

| Обычно в ТЗ | Что в шаблоне |
|-------------|---------------|
| Регистрация с полями и проверками | `/register/` |
| Вход по логину и паролю | `/login/` |
| Пользователь создаёт «заявку / заказ / запись» | `/applications/new/` |
| Пользователь видит свои записи | `/applications/` |
| Админ видит всё и меняет статус | `/admin-panel/` |
| Отзыв после завершения услуги (модуль 2) | форма отзыва на странице заявок |

### Нужно дорабатывать, если:

- другие поля регистрации (паспорт, ИНН, дата рождения…);
- вместо «заявки на курс» — «бронь номера», «заказ детали», «запись к врачу»;
- больше таблиц (каталог товаров, расписание, категории…);
- другие статусы («Ожидает оплаты», «Отменена»…);
- админ с другим логином/паролем.

**Это нормально** — вы меняете модели, формы и тексты. Каркас (Django + страницы + админка) остаётся.

## 2. Что одинаково почти во всех заданиях

Почти любое задание ДЭ по модулю 1 можно разложить так:

```
┌─────────────┐     регистрация      ┌──────────────┐
│  Гость      │ ──────────────────►  │ Пользователь │
└─────────────┘                      └──────┬───────┘
                                            │
                                     создаёт заявку
                                            ▼
                                     ┌──────────────┐
                                     │   Заявка     │◄──── админ меняет статус
                                     └──────┬───────┘
                                            │
                                     (после завершения)
                                            ▼
                                     ┌──────────────┐
                                     │   Отзыв      │  ← необязательно в модуле 1
                                     └──────────────┘
```

В шаблоне это три таблицы:

| Таблица в БД | Файл | За что отвечает |
|--------------|------|-----------------|
| `portal_user` | `portal/models.py` → `User` | Регистрация, вход |
| `portal_application` | `portal/models.py` → `Application` | Главная сущность задания |
| `portal_review` | `portal/models.py` → `Review` | Отзывы (модуль 2) |

**Переименовывать таблицы не обязательно** — важно, чтобы **поля внутри** соответствовали заданию, а на сайте были правильные **русские названия** (в формах и шаблонах).

## 3. Алгоритм на экзамене: с чего начать

### Шаг 1. Прочитайте задание и выпишите на листок

Ответьте на 5 вопросов:

1. **Как называется система?** (для `SITE_NAME`)
2. **Какие поля при регистрации?** (логин, пароль, ФИО, телефон, email — или другие)
3. **Что пользователь создаёт?** (заявка на курс, бронь, заказ…)
4. **Какие поля у этой сущности?** (название, дата, способ оплаты…)
5. **Логин/пароль админа и статусы?** (для `exam_config.py` и модели)

### Шаг 2. Нарисуйте схему БД на бумаге

Прямоугольники = таблицы, стрелки = связи.  
Пример для «Корочки»:

```
User ──< Application >── Review (один отзыв на заявку)
```

### Шаг 3. Сравните с тем, что уже есть в шаблоне

| Нужно в задании | Уже есть в шаблоне? | Действие |
|-----------------|---------------------|----------|
| То же самое | ✅ | Меняете только `exam_config.py` |
| Другое название поля | почти | Переименовать в `models.py` + формы + шаблоны |
| Новое поле | ❌ | Добавить в `models.py` → migrate |
| Новая таблица | ❌ | Новый класс в `models.py` → migrate → новая страница |
| Лишнее поле | — | Удалить из модели (на экзамене лучше не добавлять лишнего) |

### Шаг 4. Меняйте файлы по порядку (см. раздел 6)

### Шаг 5. `makemigrations` → `migrate` → проверка в браузере

## 4. Как придумать таблицы БД из текста задания

### Правило экзамена

> «Достаточно создать таблицы, поля и связи, **непосредственно относящиеся к подсистеме**»

Не нужно моделировать всю фирму целиком — только то, что видно на сайте.

### Как читать формулировки ТЗ

| Фраза в задании | Что это в БД |
|-----------------|--------------|
| «Пользователь вводит логин, пароль, ФИО…» | поля таблицы `User` |
| «Данные заносятся в базу» | `CharField`, `DateField`, `EmailField`… |
| «Все заявки хранятся в БД» | отдельная таблица `Application` (или своё имя класса) |
| «Заявка привязана к пользователю» | `ForeignKey` на `User` |
| «Админ видит все заявки» | админ читает всю таблицу заявок |
| «Статус: Новая / В работе / Завершено» | поле `status` с `TextChoices` |
| «Оставить отзыв о услуге» | таблица `Review` + связь с заявкой |

### Типы полей — шпаргалка

| Что хранить | Тип в Django | Пример |
|-------------|--------------|--------|
| Текст (короткий) | `CharField(max_length=…)` | название курса, марка авто |
| Длинный текст | `TextField()` | отзыв, комментарий |
| Дата | `DateField()` | дата начала обучения |
| Дата и время | `DateTimeField()` | когда создан заказ |
| Число | `IntegerField()`, `PositiveSmallIntegerField()` | оценка 1–5 |
| Да/нет | `BooleanField()` | согласие на рассылку |
| Список вариантов | `CharField(choices=…)` | способ оплаты, статус |
| Связь «много к одному» | `ForeignKey` | у пользователя много заявок |
| Связь «один к одному» | `OneToOneField` | один отзыв на заявку |

## 5. Три уровня изменений

### Уровень 1 — только тексты (5–15 минут)

**Когда:** та же предметная область, те же поля, другое название фирмы или кнопок.

**Меняете только:** `exam_config.py` + картинки в `static/images/` + цвета в CSS.

Пример: «Корочки.есть» → «Учебный центр Профи» — те же поля, другие названия курсов и логин админа.

---

### Уровень 2 — другие поля, та же структура (30–60 минут)

**Когда:** другая сфера, но логика та же: регистрация → заявка → админ → отзыв.

**Меняете:**

1. `exam_config.py` — тексты, списки, валидация, админ
2. `portal/models.py` — поля в `User` и `Application`
3. `portal/forms.py` — какие поля на формах
4. `portal/validators.py` — регулярки
5. `portal/templates/portal/` — подписи на страницах
6. `portal/views.py` — если меняется логика (когда можно оставить отзыв)
7. Команды: `makemigrations` + `migrate`

**Не переименовывайте классы** `User`, `Application`, `Review` без необходимости — так меньше файлов трогать. Меняйте **русские названия** через `verbose_name` и шаблоны.

---

### Уровень 3 — новые сущности (1–2 часа)

**Когда:** в задании каталог услуг, несколько типов записей, отдельная таблица «Курсы» в БД.

**Пример:** курсы хранятся в таблице `Course`, а в заявке — выбор из списка через `ForeignKey`.

Добавляете в `models.py`:

```python
class Course(models.Model):
    title = models.CharField("Название", max_length=255)

class Application(models.Model):
    course = models.ForeignKey(Course, on_delete=models.CASCADE)  # вместо course_name
    # ...
```

Потом: migrate, правка форм, возможно `admin.py` и шаблонов.

## 6. Какие файлы менять и в каком порядке

Рекомендуемый порядок (не перепрыгивайте):

```
1. exam_config.py      ← тексты, списки, логин админа, регулярки
2. portal/models.py    ← структура БД (главное!)
3. makemigrations + migrate
4. portal/validators.py
5. portal/forms.py
6. portal/views.py
7. portal/templates/
8. portal/admin.py     ← для django-admin и ER
9. static/css/       ← дизайн под новую фирму
```

### Связь «файл → за что отвечает»

| Файл | Что меняется |
|------|--------------|
| `exam_config.py` | Название сайта, курсы в списке, логин/пароль админа, тексты кнопок, регулярки |
| `portal/models.py` | **Таблицы и столбцы БД** — главный файл для другой сферы |
| `portal/forms.py` | Какие поля видит пользователь на экране |
| `portal/validators.py` | Проверка логина, телефона, ФИО по ТЗ |
| `portal/views.py` | Кто что видит, когда можно отзыв, фильтры админа |
| `portal/urls.py` | Адреса страниц (если добавляете новые разделы) |
| `portal/templates/portal/*.html` | Внешний вид и подписи |
| `portal/admin.py` | Список полей в Django Admin |
| `static/css/main.css` | Цвета портала |
| `static/css/admin.css` | Цвета админ-панели |

## 7. Пример: было «Корочки» → стало «Автопрокат»

### Текст задания (выдуманный пример)

> Портал «АвтоРент». Регистрация: логин, пароль, ФИО, телефон, email.  
> Клиент бронирует автомобиль: марка/модель (из списка), дата выдачи, способ оплаты (карта / наличные).  
> Админ: логин `AdminRent`, пароль `Auto2026`. Статусы: «Новая», «Авто выдано», «Аренда завершена».

### Что меняем

#### 1. `exam_config.py`

```python
SITE_NAME = "АвтоРент"
SITE_TAGLINE = "Прокат автомобилей"
ADMIN_LOGIN = "AdminRent"
ADMIN_PASSWORD = "Auto2026"

COURSE_CHOICES = [  # можно переименовать в CAR_CHOICES в коде
    "Toyota Camry",
    "Hyundai Solaris",
    "Kia Rio",
]
```

#### 2. `portal/models.py` — класс Application

Меняем **подписи** и при необходимости **имена полей**:

```python
class Application(models.Model):
    class PaymentMethod(models.TextChoices):
        CARD = "card", "Банковская карта"
        CASH = "cash", "Наличными"

    class Status(models.TextChoices):
        NEW = "new", "Новая"
        IN_PROGRESS = "in_progress", "Авто выдано"
        COMPLETED = "completed", "Аренда завершена"

    user = models.ForeignKey(User, on_delete=models.CASCADE, related_name="applications")
    course_name = models.CharField("Марка и модель", max_length=255)  # то же поле, другая подпись
    start_date = models.DateField("Дата выдачи автомобиля")
    payment_method = models.CharField("Способ оплаты", max_length=20, choices=PaymentMethod.choices)
    status = models.CharField("Статус", max_length=20, choices=Status.choices, default=Status.NEW)
    # ...
```

> **Совет:** поле можно оставить `course_name` в коде, а на сайте показывать «Марка и модель» — так меньше правок в формах. На экзамене это допустимо, если **на экране** всё соответствует ТЗ.

Или переименовать в `car_model` — тогда правите ещё `forms.py`, шаблоны и делаете **миграцию переименования**.

#### 3. `portal/forms.py`

Обновите подписи и списки (`COURSE_CHOICES` → машины из `exam_config.py`).

#### 4. Шаблоны

В `applications/list.html`, `create.html` — замените слова «курс» → «автомобиль», «обучение» → «аренда».

#### 5. БД

```bash
python manage.py makemigrations portal
python manage.py migrate
```

## 8. Как правильно создать/обновить БД (команды)

### Первый запуск (база пустая)

```bash
python manage.py makemigrations portal
python manage.py migrate
```

Появится файл `db.sqlite3` — это вся база.

### Вы изменили `models.py` (добавили/убрали поле)

```bash
python manage.py makemigrations portal
python manage.py migrate
```

Django создаст файл в `portal/migrations/` и применит изменения к БД.

### На экзамене: база с нуля в новый день

По условию экзамена БД проектируете **сами в день сдачи**. Алгоритм:

1. Прочитали ТЗ.
2. Поправили `models.py` под задание.
3. `makemigrations` → `migrate`.
4. Проверили в браузере: регистрация → заявка → админ.
5. Сделали ER-диаграмму в СУБД.

### Если migrate ругается на старые данные

**Дома (данные не жалко):**

```bash
# Windows
del db.sqlite3
# Linux
rm db.sqlite3

python manage.py migrate
```

**На экзамене:** лучше не удалять БД в конце — сначала `makemigrations`, смотрите что Django предлагает.

### PostgreSQL на сервере экзамена

1. Создайте БД в pgAdmin / через `createdb`.
2. Пропишите доступ в `config/settings.py`.
3. `pip install psycopg2-binary`
4. `python manage.py migrate`

## 9. ER-диаграмма под новое задание

Критерий **Д2Д1**: диаграмма должна **совпадать** с реальной БД.

### Что показать на диаграмме

Минимум для шаблона:

```
┌─────────────────┐       ┌─────────────────────┐       ┌─────────────────┐
│  portal_user    │       │ portal_application  │       │  portal_review  │
├─────────────────┤       ├─────────────────────┤       ├─────────────────┤
│ id              │──┐    │ id                  │──┐    │ id              │
│ username        │  └───►│ user_id (FK)        │  └───►│ application_id  │
│ password        │       │ course_name         │       │ text            │
│ full_name       │       │ start_date          │       │ rating          │
│ phone           │       │ payment_method      │       │ created_at      │
│ email           │       │ status              │       └─────────────────┘
│ ...             │       │ created_at          │
└─────────────────┘       └─────────────────────┘
```

Если добавили поля — **добавьте их на диаграмму**.

### Где сделать

| Программа | Как |
|-----------|-----|
| **pgAdmin** | ПКМ на БД → ERD For Database |
| **DBeaver** | ER-диаграмма по подключению |
| **draw.io** | Вручную по таблицам из `models.py` |

### Важно для оценки

- Каждый столбец из `models.py` должен быть на диаграмме.
- Каждый столбец на диаграмме должен использоваться на сайте (критерий Б2Д1 — 100% использование полей БД).

## 10. Чеклист перед сдачей

### Соответствие ТЗ

- [ ] Название портала как в задании (`exam_config.py`)
- [ ] Все поля регистрации из ТЗ есть и проверяются
- [ ] Сообщения об ошибках при неверном входе
- [ ] Ссылка «Регистрация» ↔ «Вход»
- [ ] Пользователь видит **только свои** заявки
- [ ] Админ видит **все** заявки
- [ ] Логин/пароль админа **точно** как в задании (регистр букв!)
- [ ] Статусы совпадают с ТЗ
- [ ] Отзыв только когда услуга завершена (если есть в модуле 2)

### База данных

- [ ] `models.py` соответствует заданию
- [ ] Выполнены `makemigrations` и `migrate`
- [ ] ER-диаграмма совпадает с таблицами
- [ ] Все поля таблиц используются на сайте

### Дизайн (модуль 2)

- [ ] Цвета подходят под сферу (не оставили «чужие» от Корочек без смысла)
- [ ] Картинки из приложения к заданию
- [ ] Работает на ширине 390px (телефон)
